## 📊 一、数据库表结构设计（SQL）

**建议采用**: 核心表 + 扩展 JSONB + 关联表 的模式

### 1. 职位主表（jobs）

```sql
CREATE TABLE jobs (
    id SERIAL PRIMARY KEY,
    job_id_platform VARCHAR(100) NOT NULL,  -- 平台原始 ID
    platform VARCHAR(20) NOT NULL,          -- boss, zhilian, lagou, 51job
    title VARCHAR(255) NOT NULL,            -- 职位名称
    company_name VARCHAR(255) NOT NULL,     -- 公司全称
    city VARCHAR(50) DEFAULT '深圳',
    district VARCHAR(50),                   -- 比如：宝安区、南山区
    salary_min INT,                         -- 薪资下限（单位：元）
    salary_max INT,                         -- 薪资上限
    experience_req VARCHAR(50),             -- 经验要求
    education_req VARCHAR(50),              -- 学历要求
    tags TEXT[],                            -- 平台自带标签
    jd_raw TEXT NOT NULL,                   -- 原始 JD 文本（用于对比或重新解析）
    jd_analysis JSONB,                      -- AI 解析后的结构化数据 (关键！)
    unique_hash CHAR(64) UNIQUE,            -- 用于去重 (SHA-256)
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

### 2. 投递记录表（applications）

```sql
CREATE TABLE applications (
    id SERIAL PRIMARY KEY,
    job_id INTEGER REFERENCES jobs(id),
    resume_version VARCHAR(50),             -- 投递时使用的简历版本
    match_score NUMERIC(4, 2),              -- 5 维评分系统得分
    match_reason TEXT,                      -- AI 给出的匹配理由简述
    status VARCHAR(20) DEFAULT 'pending',   -- pending, applied, rejected, interview
    applied_at TIMESTAMP WITH TIME ZONE,
    platform_response TEXT,                 -- 平台返回的投递结果快照
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
```

### 3. 索引优化

```sql
CREATE INDEX idx_jobs_city_salary ON jobs(city, salary_min, salary_max);
CREATE INDEX idx_jobs_jd_analysis ON jobs USING GIN (jd_analysis);  -- 针对 JSONB 的倒排索引
CREATE INDEX idx_unique_hash ON jobs(unique_hash);
```

## 🔍 二、核心问题深度解答

### Q1: 如何设计去重机制？

**建议方案**: MD5/SHA-256 复合哈希去重

**原因**: 单纯靠平台 JobID 不可靠（同一职位可能在不同平台发布，或同一平台重复发布）

**计算公式**:
```python
unique_hash = hash(company_name + title + city + salary_min + salary_max)
```

**实现逻辑**:
```sql
-- 在写入前计算该 Hash，利用 SQL 的 ON CONFLICT 自动过滤重复数据
INSERT INTO jobs (..., unique_hash) 
VALUES (..., 'abc123')
ON CONFLICT (unique_hash) DO NOTHING;
```

**优势**: 即便 JD 描述略有改动，只要核心信息一致，就不会造成数据堆积。

---

### Q2 & Q3: JSONB 存储 AI 解析结果

**结论**: 非常适合！

**原因**: 品牌策划岗位的技能点（如：全案策划、新媒体运营、小红书投放）非常细碎。

**存储建议**:
```json
{
  "skills": ["全案策划", "新媒体运营", "小红书投放"],
  "industry_tags": ["互联网", "电子商务"],
  "key_requirements": ["3-5 年经验", "本科及以上学历"],
  "subway_station": "后海站",
  "match_details": {
    "skill_match": 0.8,
    "experience_match": 0.9,
    "industry_match": 0.7
  }
}
```

**查询优势**: PostgreSQL 的 JSONB 支持 GIN 索引，可以实现毫秒级的包含查询。

**示例查询**:
```sql
-- 查找所有需要"小红书运营"技能的深圳职位
SELECT * FROM jobs 
WHERE city = '深圳' 
  AND jd_analysis->'skills' ? '小红书运营';
```

---

### Q4: 投递记录表的关系

**关系**: One-to-Many（一个职位可以被多次分析，但通常对应一次投递）

**设计要点**:
> 务必在 `applications` 表中固化投递时的 `match_score`。因为你的简历和算法在更新，但历史投递的评分应作为回测数据保留。

---

### Q5: 性能优化建议

| 优化项 | 建议 | 说明 |
|--------|------|------|
| **索引** | 对 `salary` 和 `city` 建立复合索引 | 加速薪资 + 城市筛选 |
| **分区 (Partitioning)** | 按 `created_at` 进行月度分区 | 数据量接近 10 万 + 时使用 |
| **分区优势** | `Drop Partition` 代替 `Delete` | 清理过期职位数据性能更高 |
| **全文检索** | 配合 `tsvector` 功能 | 对 `jd_raw` 进行模糊搜索 |

## 🛠️ 三、最佳实践建议（针对深圳求职 Agent）

### 1. 地理围栏细化

**原因**: 深圳通勤压力大

**建议**: 在 `jd_analysis` 中增加 `subway_station` 字段

**示例**:
```json
{
  "subway_station": "后海站",
  "commute_time": 30
}
```

**AI 解析**: JD 中提到"后海站"、"固戍站"时直接提取，作为匹配算法的加分项。

---

### 2. API 与 Browser 降级策略

| 方式 | 用途 | 说明 |
|------|------|------|
| **API 抓取** | 获取 `job_id` 和基础字段 | 快速、高效 |
| **Browser 抓取** | 获取完整上下文 | 利用 `browser.snapshot()` 获取隐藏的"公司福利"或"面试要求" |

---

### 3. 5 维评分系统落库

**建议**: 将 5 维评分的明细存入 JSONB

**示例**:
```json
{
  "match_score": 0.85,
  "match_details": {
    "skill_match": 0.8,
    "experience_match": 0.9,
    "industry_match": 0.7,
    "salary_match": 0.85,
    "city_match": 1.0
  }
}
```

**优势**: 方便后续调整算法权重，支持回测分析。
